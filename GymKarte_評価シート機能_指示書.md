# GymKarte 評価シート機能 実装指示書

> Cowork 引き継ぎ用 / この指示書を受領後、ただちに実装に着手すること。
> 不明点があれば実装を止めず、本書末尾の「判断に迷った場合の既定値」に従って進める。

---

## 0. 一言サマリ

GymKarte の会員詳細に **「評価シート」タブ** を新設する。
会員ごとに、毎月の **基準種目の重量・回数（パフォーマンス）の伸びをグラフで可視化** し、
トレーナーが手書き相当のフィードバックを添えて、**ワンクリックで綺麗に印刷** できるようにする。
シートは **月次で発行・保存** し、**過去の発行分をいつでも閲覧** できる履歴機能を持たせる。

この機能は「一般会員（ポッププラン）向け特典」として配布する評価用紙のデジタル基盤である。
目的は (1) 月1回の来店動機づくり、(2) 成長実感による継続率向上、(3) 月額プランの所属感醸成。

---

## 1. 技術前提（既存スタックに合わせること）

- **必ず既存の GymKarte コードベース内に実装する**（新規プロジェクトを立てない）
- Electron + React + SQLite + Tailwind CSS
- 既存の会員管理・SVGボディマップ・回数券・デイリーログと同じ設計思想・命名規則・ディレクトリ構成を踏襲する
- 既存の会員テーブルと **会員ID で紐付ける**（外部キー）
- グラフ描画ライブラリは軽量なものを使用（推奨: **Recharts**。既に別ライブラリが入っていればそれに合わせる）
- 印刷は **CSS の `@media print` + ブラウザ印刷** で実現する（PDF生成ライブラリは必須ではないが、レイアウト崩れを防げるなら採用可）

---

## 2. データモデル（SQLite）

### 2-1. 評価シート本体テーブル `evaluation_sheets`

| カラム | 型 | 説明 |
|--------|----|----|
| `id` | INTEGER PK AUTOINCREMENT | シートID |
| `member_id` | INTEGER FK | 会員ID（既存会員テーブル参照） |
| `year_month` | TEXT | 発行対象月。`YYYY-MM` 形式（例 `2026-06`） |
| `issued_at` | TEXT (ISO8601) | 発行日時 |
| `trainer_name` | TEXT | 担当トレーナー名 |
| `feedback_positive` | TEXT | **必須**: 必ず前向きに触れるコメント欄 |
| `feedback_next` | TEXT | 次月の目標・アドバイス欄 |
| `created_at` / `updated_at` | TEXT | 監査用 |

- `member_id` + `year_month` で **UNIQUE 制約**（同一会員・同一月のシートは1枚）
- 同月に再発行する場合は **上書き更新**（履歴の重複を防ぐ）

### 2-2. 種目別記録テーブル `evaluation_records`

| カラム | 型 | 説明 |
|--------|----|----|
| `id` | INTEGER PK | |
| `sheet_id` | INTEGER FK | `evaluation_sheets.id` 参照 |
| `exercise_key` | TEXT | 基準種目の識別キー（下記の固定マスタ参照） |
| `weight` | REAL | 重量（kg）。種目により null 可 |
| `reps` | INTEGER | 回数。種目により null 可 |
| `seconds` | INTEGER | 秒数（プランク等）。種目により null 可 |
| `note` | TEXT | 種目ごとの備考（任意） |

### 2-3. 基準種目マスタ（固定）

種目は **トレーナー3名が誰でも同じ項目を測れるよう固定** する。
初期マスタは以下5項目（コード内に定数として定義。将来 DB 化してもよいが今回はハードコードで可）:

| key | 表示名 | 記録単位 |
|-----|--------|---------|
| `squat` | スクワット | 重量(kg) × 回数 |
| `chest_press` | チェストプレス | 重量(kg) × 回数 |
| `lat_pulldown` | ラットプルダウン | 重量(kg) × 回数 |
| `plank` | プランク | 秒数 |
| `row` | ローイング | 重量(kg) × 回数 |

> ※種目名・項目数は後で運用しながら調整したい。**マスタを1箇所にまとめ、変更が容易な構造**にしておくこと。

## 3. UI 要件

### 3-1. タブの追加

- 会員詳細画面に既存タブ（基本情報 / ボディマップ / 回数券 / デイリーログ等）と並べて **「評価シート」タブ** を追加
- タブ内は上から: ①月選択・新規発行ボタン → ②グラフ → ③当月の記録入力フォーム → ④フィードバック入力 → ⑤印刷ボタン → ⑥過去シート一覧

### 3-2. 記録入力フォーム

- 基準種目ごとに、重量・回数（または秒数）を入力できるシンプルなフォーム
- **前回値をプレースホルダーまたは薄字で表示** し、伸びを意識しながら入力できるようにする
- 入力途中の保存（下書き）と、確定（発行）を分ける

### 3-3. フィードバック欄

- `feedback_positive`（必ず褒める欄）は **必須入力**。空欄では発行できないバリデーションを入れる
  - ※運用ルール「数字が伸びない月でも必ず前向きに触れる」を仕組みで担保するため
- `feedback_next`（次月の目標）は任意

---

## 4. グラフ要件（最重要）

- **種目ごとに、月推移の折れ線グラフ** を表示する
- X軸 = 発行月（時系列）、Y軸 = 重量 または 回数 または 秒数
- 重量×回数の種目は、**重量の推移**を主軸とし、回数は補助表示（ツールチップ or 第二系列）
- データ点が1つしかない月初期でも破綻しない描画にする
- **「先月より伸びた項目」が一目で分かる**演出を入れる（例: 上昇は緑、停滞・下降はグレー等）
- ポップ層は初心者寄りで毎月数字が伸びやすい。**成長が映えるよう、見やすさ最優先**で設計する

---

## 5. 印刷要件（ワンクリック・レイアウト固定）

最も重視する要件。**印刷ボタン1つで、毎回同じ綺麗なレイアウトで出力**できること。

- 用紙: **A4 縦** 想定
- `@media print` 専用スタイルを用意し、画面UI（ボタン・タブ・サイドバー等）は印刷時に非表示
- 印刷レイアウトに含める要素:
  1. ヘッダー: ジムロゴ + **マスコット「半ちゃん」** + 「○○様 評価シート（YYYY年MM月）」
  2. 基準種目ごとの当月記録（表形式、前月比を併記）
  3. 推移グラフ（印刷でも潰れないサイズ・解像度）
  4. トレーナーからのフィードバック（`feedback_positive` / `feedback_next`）
  5. フッター: 発行日・担当トレーナー名
- **1ページに収まる**ことを最優先。収まらない場合は要素サイズを調整
- ブランドイメージ（気軽・楽しい・運動習慣づくり）に合う、**親しみやすく見やすいデザイン**。堅い書類感を避ける
- 半ちゃんのトーンで、印刷シートにひとこと添えられる固定文 or 自由入力欄があると尚良い

---

## 6. 履歴・保存要件

- 毎月発行するため、**過去の発行シートをすべて保存**し、いつでも閲覧・再印刷できる
- 評価シートタブ内に **過去シート一覧**（発行月の降順リスト）を設置
- 一覧の各行から、その月のシートを開いて閲覧・再印刷できる
- 過去シートは原則 **閲覧専用**（編集は当月のみ。過去分の修正可否は要確認 → 既定では編集可だが「修正済み」が分かるようにする）
- グラフは過去全月のデータを通して描画されるため、**履歴が蓄積するほどグラフの価値が上がる**設計

---

## 7. 完了の定義（受け入れ基準）

以下がすべて満たされたら完了とする。

1. 会員詳細に「評価シート」タブが表示される
2. 基準5種目の重量・回数・秒数を入力し、当月シートとして保存・発行できる
3. `feedback_positive` 未入力では発行できない
4. 種目ごとの月推移グラフが表示され、前月比の伸びが視覚的に分かる
5. 印刷ボタン1つで、A4縦・1ページ・崩れないレイアウトで出力できる
6. 同一会員・同一月の重複シートが作られない（UNIQUE制約が効く）
7. 過去の発行シートを一覧から開いて閲覧・再印刷できる
8. 既存機能（会員管理・ボディマップ・回数券・デイリーログ）に影響を与えていない

---

## 8. 判断に迷った場合の既定値

- グラフライブラリが未導入 → **Recharts** を採用
- 過去シートの編集可否 → **編集可。ただし `updated_at` を更新し、再印刷時も最新内容を反映**
- 印刷方式 → まず `@media print` + ブラウザ印刷で実装。レイアウトが安定しなければ PDF 生成を検討
- 種目マスタの持ち方 → 今回は**定数としてハードコード**（1ファイルに集約し変更容易に）
- デザインの方向性 → 「気軽・楽しい・運動習慣づくり」に寄せ、堅い書類感を避ける

---

## 9. 実装の進め方（推奨順序）

1. データモデル（テーブル作成・マイグレーション）
2. 当月記録の入力フォーム + 保存/発行ロジック（UNIQUE制約・バリデーション込み）
3. グラフ表示（履歴データを時系列で描画）
4. 過去シート一覧・閲覧
5. 印刷レイアウト（最後に時間をかけて詰める）
6. 既存機能への回帰確認

---

*本機能は会員特典の中核。「成長が見える」「毎月もらえる」体験の質がそのまま継続率に効くため、特にグラフの見やすさと印刷レイアウトの完成度を重視すること。*
